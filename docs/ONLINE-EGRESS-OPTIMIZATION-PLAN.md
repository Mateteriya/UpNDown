# План оптимизации egress Supabase и стабильности онлайн-стола

Цель: снизить исходящий трафик с проекта (PostgREST / Realtime), не ломая возврат в партию после F5 и без систематических «залипаний» стола.

## Текущая картина

- Периодический опрос `getRoomForSyncPoll` в лобби и в партии компенсирует неполную доставку Realtime.
- Каждый тик тянет строку `game_rooms` с JSON `game_state` — основной вклад в egress.
- Дублирующиеся снимки при той же ревизии уже частично отфильтровываются; конфликты ревизий дают повторные записи и визуальные артефакты («мигание»).

## Этап 1 — измерение (короткий)

1. В DevTools зафиксировать средний размер ответа `game_rooms` select и число запросов в минуту на одного клиента в **waiting** и в **playing**.
2. В Dashboard Supabase сверить egress с этими оценками (порядок величин должен сходиться).

## Этап 2 — «лёгкий» тик без полного JSON

1. Добавить чтение только полей `id`, `game_state_revision`, `updated_at` (view или RPC), вызываемое с текущим интервалом опроса или реже.
2. Если `revision` не изменился — не делать полный `getRoom` / не переприменять состояние (уже частично есть через ревизию; довести до одного правила).
3. Если `revision` изменился — один полный fetch (как сейчас).

**Ожидаемый эффект:** кратное снижение трафика в фазах без ходов (ожидание оппонента, пауза).

## Этап 3 — адаптивные интервалы опроса

1. **Базовый** период в `playing` увеличить (например с ~760 ms до 1.2–1.5 s) при «здоровом» Realtime (последнее событие/успешный apply недавно).
2. После локального `send*` — короткий **burst** опроса 300–500 ms, 3–5 тиков, затем снова базовый период.
3. При ошибке сети / пустом ответе — **backoff** (1.5s → 3s → max), плюс уже существующий resync.
4. В **waiting** лобби — отдельный более длинный период (слоты редеют не каждые сотни мс).

**Риск:** редкое отставание UI; mitigated burst-ом после собственных действий и кнопкой resync.

## Этап 4 — уменьшение полезной нагрузки `game_state`

1. Аудит: что обязательно в hot-path синхронизации, а что можно вынести (например длинная история раздач только для таблицы результатов).
2. При необходимости — вынести тяжёлые поля в отдельные таблицы / догружать по запросу.

## Этап 5 — согласованность и мигание

1. Свести правила: один источник «применять снимок / игнорировать» по паре `(game_state_revision, updated_at)`.
2. Не сбрасывать локальные анимации карт при идентичном логическом состоянии (глубокое сравнение или хэш критичных полей).

## Этап 6 — инфраструктура (по необходимости)

- Сжатие на уровне CDN не снижает egress Supabase; выигрыш даёт **меньше байт в ответе** и **меньше ответов**.
- Самостоятельный VPS снимает биллинг Supabase, но не убирает необходимость бережно относиться к полезной нагрузке.

## Порядок внедрения

Рекомендуемый порядок: **2 → 3 → 4** (сначала дешёвый тик и адаптив, потом «диета» JSON). Этап 5 можно вести параллельно с 3, если мигание мешает отладке.

## Критерии готовности

- Egress в день теста с 4 людьми в партии — заметно ниже при субъективно той же отзывчивости.
- После F5 в **playing** — по-прежнему гарантированный возврат в партию.
- Нет роста частоты `conflict` / рассинхрона, измеряемого логами или отзывами.

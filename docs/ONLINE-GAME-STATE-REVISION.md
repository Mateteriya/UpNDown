# Ревизия `game_state` и проверка синхронизации

## 1. Что это значит простыми словами

В таблице **`game_rooms`** в Supabase появляется число **`game_state_revision`**. Оно **увеличивается только когда меняется поле `game_state`** (ход, торги, новая раздача и т.д.). Если обновили **только слоты** (имя, аватар, кто в комнате) — **`updated_at` меняется, а ревизия нет**.

Клиент тогда понимает: «это то же состояние стола, что уже было» — и **не подменяет** раздачу устаревшим JSON из-за «свежего» времени строки.

**Пока миграция не выполнена**, в ответе API поля ревизии нет — приложение работает по **старой ветке** (эвристики по фазам/конуру и т.д.). Играть можно, но поведение хуже, чем после миграции.

## 2. Как применить миграцию (куда зайти)

### Вариант A — через сайт Supabase (чаще всего)

1. Открой [supabase.com](https://supabase.com) → **свой проект** (тот же, что в `VITE_SUPABASE_URL`).
2. Слева: **SQL Editor** → **New query**.
3. Открой в репозитории файл  
   `supabase/migrations/20250401120000_game_rooms_state_revision.sql`  
   и **скопируй весь текст целиком** в окно запроса.
4. Нажми **Run** (или Ctrl+Enter).
5. Должно быть **Success** (без ошибки).

**Проверка:** **Table Editor** → таблица **`game_rooms`** → должна появиться колонка **`game_state_revision`** (число, по умолчанию `0` у старых строк).

### Вариант B — Supabase CLI (если проект уже связан с папкой)

В корне репозитория (где есть `supabase/`):

```bash
supabase db push
```

или вручную выполни тот же SQL, что в файле миграции.

### Если ошибка про `EXECUTE PROCEDURE`

На части версий PostgreSQL для триггера пишут **`EXECUTE FUNCTION`** вместо **`EXECUTE PROCEDURE`**. Замени в конце файла миграции одну строку:

```sql
execute function public.game_rooms_bump_state_revision();
```

и выполни скрипт снова.

## 3. Ручной чеклист (инварианты)

Прогони после миграции на **двух устройствах/браузерах** (хост + гость). Галочка = ожидаемое поведение.

| Сценарий | Ожидание |
|----------|----------|
| Оба в лобби, гость заходит | Хост видит гостя в слотах без «пропажи» стола. |
| Хост нажимает «Начать игру» | У всех одновременно раздача; нет долгой «пустой» раздачи с последующей подменой карт. |
| Два заказа подряд (разные игроки) | У второго клиента заказы не отстают; нет отката к одному заказу. |
| Ход по очереди (3–4 хода взятки) | Карта не «отскакивает» назад в руку; при опоздании Realtime догоняет актуальным `game_state`. |
| Один клиент F5 во время партии | После перезагрузки тот же стол и тот же ход (в пределах сессии/комнаты). |
| Вкладка свернута / другой экран 30–60 с | После возврата состояние подтягивается (poll fallback + `visibility` refresh). |
| Ручная пауза (слот → ИИ) и возврат | Слоты и стол согласованы; после возврата — снова свой ход/имя. |
| Смена аватара/имени в лобби | Не сбрасывает и не «откатывает» уже начатую раздачу у других. |

Если что-то падает — зафиксируй: **фаза**, **кто ходил**, **один клиент или два**, **была ли миграция** (колонка в Table Editor есть или нет).

## 4. Нужно ли «сужать legacy» в коде

**Сейчас:** после миграции основная логика — по **`game_state_revision`**. Ветка **legacy** остаётся для БД **без** колонки и как запасной путь.

**Имеет смысл урезать legacy в коде** только когда ты уверена, что **все** окружения (прод, превью, локальный Supabase) уже с миграцией. До тех пор чеклист выше важнее, чем удаление `if` в `OnlineGameContext`.

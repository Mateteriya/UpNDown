# Когда техдиректор поднял WebSocket: ваши шаги

**Только для владельца продукта.**  
Предполагается, что техдиректор выполнил [TECH-DIRECTOR-ONLINE-SERVER.md](./TECH-DIRECTOR-ONLINE-SERVER.md) и передал вам **`wss://…`**.

LAN-вечеринки по Wi‑Fi **не меняются** — по-прежнему `npm run host:app` на вашем ПК. Этот документ про **облачный онлайн** (игра через интернет).

---

## 1. Что вы должны получить от техдиректора

| Данные | Пример | Зачем |
|--------|--------|--------|
| **URL WebSocket** | `wss://game.up2down.online` | Главная настройка фронта |
| **Health URL** | `https://game.up2down.online/api/version` | Проверка, что сервер жив |
| (Опционально) команда рестарта | `systemctl restart updown-ws` | На случай сбоя |
| (Опционально) поле `build` из `/api/version` | `host-panel-2026-06-06-installer` | Понять, какая версия кода на VPS |

**Проверьте сами до настройки фронта:**

```powershell
curl https://game.up2down.online/api/version
```

В ответе: `"hostPanel": true` (или похожий JSON). Если пусто или ошибка TLS — вернуться к техдиректору, **не** включать WS на проде.

---

## 2. Настройка фронта (прод)

### 2.1. Переменные на хостинге (Vercel / аналог)

В настройках проекта → **Environment Variables** (Production):

| Переменная | Значение |
|------------|----------|
| `VITE_ONLINE_TRANSPORT` | `ws` |
| `VITE_WS_URL` | `wss://game.up2down.online` *(URL от техдиректора, без слэша в конце)* |

Оставьте как есть (если уже настроены):

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

Supabase **нужен** для входа в аккаунт и профилей; **партия** при `ws` идёт через сокет, не через Supabase Realtime.

### 2.2. Локальная разработка (`.env.local`)

```env
VITE_ONLINE_TRANSPORT=ws
VITE_WS_URL=wss://game.up2down.online
```

Или для отладки против локального сервера:

```env
VITE_ONLINE_TRANSPORT=ws
VITE_WS_URL=ws://192.168.1.5:3001
```

После правки `.env.local` — перезапуск `npm run dev`.

### 2.3. Деплой

После сохранения переменных на Vercel — **Redeploy** production (новый билд обязателен: Vite вшивает `VITE_*` при сборке).

---

## 3. Что **не** трогать

| Режим | Как работает | Не смешивать |
|-------|--------------|--------------|
| **LAN на вашем ПК** | `npm run host:app`, QR, `http://IP:3001/play/` | Не подставлять LAN IP в Vercel |
| **Supabase-онлайн (старый)** | `VITE_ONLINE_TRANSPORT` не задан или `supabase` | Откат, если WS сломан |
| **LAN-установщик .exe** | Отдельная загрузка с сайта (позже) | Не связан с VPS |

---

## 4. Сценарий проверки (обязательный)

Выполнить **после** деплоя фронта с `VITE_WS_URL`.

### 4.1. Два устройства из **разных сетей**

Важно: не та же Wi‑Fi, что у вас дома — иначе это повтор LAN, а не облако.

1. **Устройство A** (ПК или телефон): открыть **прод** URL приложения (PWA/Vercel).
2. Задать имя в профиле.
3. **Онлайн-лобби** → **Создать комнату**.
4. Запомнить **код** (или «Поделиться» кодом).
5. **Устройство B** (другой телефон, мобильный интернет или другая сеть): тот же прод URL → **Присоединиться** по коду.
6. Оба → **Войти в игру** → сыграть хотя бы одну полную раздачу.

### 4.2. Зал столов (если включён)

- При создании комнаты включить **«Показать в зале столов»**.
- С третьего устройства: лобби → **Зал столов** → комната видна → вход.

Флаг фронта: `VITE_PUBLIC_HALL_ENABLED` (по умолчанию включён, если не выключали).

### 4.3. Гостевая ссылка (если тестируете LAN-подобный сценарий на VPS)

На VPS обычно **нет** панели `/host` как у домашнего ПК. Гости входят **по коду** в обычном лобби приложения. QR с IP — только для LAN.

## 5. Ожидаемое поведение и известные ограничения

| Работает в WS MVP | Пока слабее / отложено |
|-------------------|-------------------------|
| Создать / войти по коду | Absent host, пауза как в Supabase |
| 3–4 игрока, ходы, заказы | Комнаты пропадают при рестарте VPS |
| Имена в лобби | Полный parity `finish_game` / рейтинг — уточнять в коде |
| Банковый режим (demo) | Чат комнаты (если был в Supabase) |
| Зал столов (`room_kind: public`) | |
| Вход **без** Google в LAN; в облаке — аккаунт по желанию | |

Если рейтинг после партии не обновился — зафиксировать баг; может потребоваться доработка A2.5 из [APP-WORKPLAN-WS-IAP-CC.md](./APP-WORKPLAN-WS-IAP-CC.md).

---

## 6. Откат на Supabase (если WS на проде сломался)

На Vercel:

1. Удалить или закомментировать `VITE_ONLINE_TRANSPORT` и `VITE_WS_URL`.
2. Redeploy.

Клиент снова пойдёт в облачный Supabase-play (с egress/polling как раньше).

LAN на вашем ПК: по-прежнему `host:app` + локальный `.env.local` с `ws://IP:3001` — **не зависит** от VPS.

## 7. Чеклист владельца продукта

- [ ] Получен `wss://…` от техдиректора
- [ ] `curl …/api/version` отвечает JSON
- [ ] `VITE_ONLINE_TRANSPORT=ws` и `VITE_WS_URL=wss://…` на **Production**
- [ ] Redeploy фронта
- [ ] Партия с двух устройств из **разных сетей**
- [ ] (Опционально) зал столов, банковая комната
- [ ] Баги записаны (чат / задачи портала)
- [ ] Команда знает: рестарт VPS = пустые комнаты

---

## 8. После успешного теста — следующие продуктовые шаги

1. **Стабилизация** 1–2 недели: reconnect после F5, имена, банк.
2. **Рейтинг / архив** после WS-партии (один вызов Supabase в конце).
3. **Снизить egress** Supabase: отключить агрессивный polling в режиме `ws` ([ONLINE-EGRESS-OPTIMIZATION-PLAN.md](./ONLINE-EGRESS-OPTIMIZATION-PLAN.md)).
4. **LAN .exe на сайт** — когда будет страница загрузки ([HOST-LAN-PUBLISH.md](./HOST-LAN-PUBLISH.md)); не блокирует облако.
5. Стратегия трёх основателей: [ROADMAP-THREE-FOUNDERS.md](./ROADMAP-THREE-FOUNDERS.md).

## 10. Кому писать при проблемах

| Симптом | Кому |
|---------|------|
| `wss://` не коннектится, 502, сертификат | Техдиректор |
| Лобби есть, но UI/ходы/банк | Разработка / AI в репо |
| Вход Google, профиль | Supabase / фронт |
| Нужен откат на Supabase-play | Вы (env на Vercel) |

## Часть A. WebSocket (игровой сервер)

**Цель:** авторитетное состояние партии на WS; Supabase — auth, профили, матч-архив, CC, лобби-коды (на переходном этапе).

### A1. Сервер (репозиторий `server/` или отдельный репо)

| # | Задача | Исполнитель | Критерий готовности |
|---|--------|-------------|---------------------|
| A1.1 | Каркас Node 18+ (Fastify/Express) + `ws` | Техдиректор | `npm run dev` поднимает процесс локально |
| A1.2 | Протокол JSON: `create_room`, `join_room`, `bid`, `play_card`, `state_sync`, `error` | Совместно | Markdown-таблица типов в `server/README.md` |
| A1.3 | Переиспользовать правила из `src/game/` (общий пакет или копия с тестами) | Владелец / техдиректор | Невалидный ход отклоняется сервером |
| A1.4 | Комнаты в памяти; TTL простоя; 3–4 слота | Техдиректор | Два клиента играют партию без рассинхрона |
| A1.5 | Reconnect по `playerToken` + `roomId` (короткий grace) | Техдиректор | F5 в течение N мин не убивает слот |
| A1.6 | Dockerfile + healthcheck | Техдиректор | `docker run` на VPS |
| A1.7 | Деплой на VPS техдиректора, `wss://` (Caddy/nginx) | Техдиректор | URL отдан владельцу продукта |
| A1.8 | Логи, restart policy, базовые метрики (uptime, rooms) | Техдиректор | Алерт при падении процесса |

### A2. Клиент (этот репозиторий)

| # | Задача | Исполнитель | Критерий готовности |
|---|--------|-------------|---------------------|
| A2.1 | `VITE_WS_URL` в `.env.example` и Vercel | Владелец | Переключение dev/prod без правки кода |
| A2.2 | `OnlineWsContext` или расширение `OnlineGameContext`: connect, dispatch, reconnect | Владелец | Лобби создаёт комнату через WS |
| A2.3 | Feature flag `VITE_ONLINE_TRANSPORT=supabase|ws` | Владелец | A/B: старый путь не ломается |
| A2.4 | Игра: состояние только из сокета; UI без изменения правил отрисовки | Владелец | Партия 4 игрока на двух устройствах |
| A2.5 | По окончании партии — **один** вызов Supabase `finish_game` + запись рейтинга | Владелец | Рейтинг и архив как сейчас |
| A2.6 | Чат комнаты: оставить Supabase или перенести позже | Отложить | Не блокирует A2.4 |

### A3. Переход и отключение polling

| # | Задача | Когда |
|---|--------|--------|
| A3.1 | Включить WS по умолчанию для новых комнат | После A2.4 стабилен 1 неделю |
| A3.2 | Снизить `ROOM_SYNC_POLL_*` или отключить в режиме `ws` | Сразу после A3.1 |
| A3.3 | Этапы 2–3 из [ONLINE-EGRESS-OPTIMIZATION-PLAN.md](./ONLINE-EGRESS-OPTIMIZATION-PLAN.md) | Только если остаётся гибрид Supabase-play |

### A4. Оценка сроков WS

| Этап | Ориентир |
|------|----------|
| A1 локально + A2.1–A2.3 | 2–3 недели |
| A2.4–A2.5 + A1.7 прод | ещё 2–3 недели |
| A3 полный переезд | +1 неделя тестов |

## Часть B. IAP и продажный слой (без gambling-лицензии)

**Цель:** монетизация через сторы / веб-платежи за **цифровые товары и доступ**, не за вывод денег.

### B1. Инфраструктура покупок

| # | Задача | Исполнитель | Критерий готовности |
|---|--------|-------------|---------------------|
| B1.1 | Выбор: **RevenueCat** (мобилка + обёртка) или нативно Google Play / App Store + отдельно Stripe для web | Владелец + международный партнёр (entity для договоров) | Решение зафиксировано в 1 абзаце |
| B1.2 | Таблица `purchases` / `entitlements` в Supabase + RPC «применить чек» | Техдиректор | Сервер не доверяет только клиенту |
| B1.3 | Webhook от RC/Stripe → Edge Function | Техдиректор | Покупка на телефоне открывает фичу на web |
| B1.4 | `productFlags` / `featureFlags`: `isPremium`, `ownedSkus[]` | Владелец | ИИ и 3 игрока читают entitlements |

### B2. UI магазина

| # | Задача | Содержание |
|---|--------|------------|
| B2.1 | Экран «Магазин» | Вкладки: Косметика · Подписка · Разовые |
| B2.2 | Косметика v1 | 2–3 темы карт + рубашка; превью на `CardView` |
| B2.3 | Подписка v1 | «Премиум»: ИИ при паузе, без рекламы (если появится), бейдж |
| B2.4 | Разовая покупка | Режим **3 игрока** (офлайн + онлайн, когда режим готов) |
| B2.5 | Восстановление покупок | Кнопка «Восстановить» (iOS/Android) |

### B3. Игровые фичи за IAP

| # | Задача | Зависимость |
|---|--------|-------------|
| B3.1 | Гейт `isPersonalAiReplacementEnabled(userId)` → entitlement | B1.4 |
| B3.2 | Режим 3 игрока: engine + UI + онлайн-слоты | Отдельный эпик; IAP после MVP режима |
| B3.3 | Звуки / стол (опционально v2) | После B2.2 |

### B4. Оценка сроков IAP

| Этап | Ориентир |
|------|----------|
| B1 + B2.1–B2.3 (одна подписка, одна тема) | 3–4 недели после старт B1.1 |
| B2.4 + B3.2 (3 игрока) | +3–5 недель (геймплей тяжелее платежей) |

## Рекомендуемый порядок (для вас лично)

```text
Недели 1–2:  A1.1–A1.4 локально + A2.1–A2.3 (параллельно B1.1 решение)
Недели 3–4:  A2.4–A2.5 + деплой A1.7
Недели 5–6:  B1.2–B2.3 (магазин + подписка + одна тема)
Недели 7–8:  C1.1–C1.5 (CC UI + hold/settle)
Недели 9+:   B3.2 (3 игрока) ИЛИ C3 (турниры) — по приоритету маркетинга
```

**WS раньше IAP?** Да, если онлайн — главный канал удержания. **IAP раньше CC?** Да: реальные деньги в сторах проще объяснить, чем виртуальная экономика.

## Чеклист «готово к бета-релизу monetization»

- [ ] Партия по WS с 4 игроками без Supabase-polling в play
- [ ] Хотя бы одна покупка (подписка или тема) проходит в sandbox стора
- [ ] Entitlements синхронизируются с аккаунтом Supabase
- [ ] CC: daily grant + комната с hold/settle на staging
- [ ] ToS / Privacy ссылаются на CC без cash-out
- [ ] `CASH_ARENA_ENABLED` остаётся `false`

## Не входит в этот план (см. roadmap)

- Cash Arena, KYC, crypto/fiat вывод
- Полная замена Supabase для лобби-кодов (можно оставить)
- Офлайн-касса Грузии (отдельный трек **а** в [ROADMAP-THREE-FOUNDERS.md](./ROADMAP-THREE-FOUNDERS.md))

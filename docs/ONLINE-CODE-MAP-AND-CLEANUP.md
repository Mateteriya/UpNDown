# Онлайн-код: карта ответственности и план приведения к порядку

Цель: не гадать, а **систематически** понять, что за что отвечает, где накладываются слои (Realtime + опрос + оптимистичные обновления + восстановление сессии), и **поэтапно** упростить без потери прод-функционала.

## 1. Где «правда» и как сравниться с продом

- **Прод**, который «уже два месяца работает», почти наверняка = ветка **`main`** (или тег/деплой с `origin/main`), **без** текущих незакоммиченных правок.
- Локально вы на ветке **`fix/deal-results-modal-freeze`** с большим диффом от `main` (~+2900 / −900 строк в 18 файлах только между коммитом и `main`; плюс **untracked** файлы: `onlineSession.ts`, `useOnlineGame.ts`, `personalAi.ts`, и т.д.).

**Команды для опоры:**

```bash
# что отличается от main (закоммиченное)
git diff main --stat
git diff main -- src/contexts/OnlineGameContext.tsx

# полная картина с рабочей директорией
git status
```

**Практический эталон:** открыть два клона/вкладки: один на `main`, один на текущей ветке — и сравнить поведение онлайна и размер ключевых файлов.

## 2. Поток данных онлайн (сверху вниз)

```
App.tsx
  └─ экран лобби / игры, восстановление сессии (?code=), модалки reclaim
LobbyScreen.tsx
  └─ create/join/leave, код комнаты, список слотов (только UI + вызовы контекста)
OnlineGameProvider (OnlineGameContext.tsx)
  └─ roomId, code, playerSlots, canonicalState, displayState (rotate)
  └─ applyRoomData / applyRoomDataOnlyIfNewer — слияние строки комнаты с сервера
  └─ Realtime subscribe + интервал опроса getRoom + refresh при visibility/online
  └─ sendBid / sendPlay / sendState / update через onlineGameSupabase
onlineGameSupabase.ts
  └─ REST: create/join/getRoom/updateRoomState, Realtime channel, leave, pause
onlineSession.ts (sessionStorage)
  └─ roomId + deviceId для «продолжить партию» после перезагрузки
GameTable.tsx
  └─ отображение стола, ход человека, хост → ходы ИИ в слотах без userId, пауза вручную
```

**Узкие места (где чаще всего «ломается»):** одновременно работают **несколько источников правды** для `canonicalState` (ваш UPDATE, чужой UPDATE, опрос, Realtime, восстановление сессии). Любая лишняя ветка «применить сервер целиком» без проверки новизны снова даёт откаты и зависания UI.

## 3. Файлы: за что отвечает каждый (чеклист для разбора)

| Файл | Зона ответственности | На что смотреть при чистке |
|------|----------------------|-----------------------------|
| `src/lib/onlineGameSupabase.ts` | Все запросы к `game_rooms`, join с retry, `updateRoomState` + `select` | Один контракт возврата (ошибка / строка комнаты), таймауты лобби |
| `src/lib/onlineSession.ts` | sessionStorage сессии комнаты | Не дублировать логику с `performSessionRestore` |
| `src/contexts/OnlineGameContext.tsx` | Единый стейт онлайна, merge, подписка, опрос | Свести к минимуму правил merge; refs только там, где нужны для гонок |
| `src/contexts/useOnlineGame.ts` | Тонкая обёртка над контекстом (Fast Refresh) | Оставить; не раздувать |
| `src/ui/LobbyScreen.tsx` | Лобби, create/join | Минимум логики; без дублирования leave/create |
| `src/ui/GameTable.tsx` | Стол, онлайн/офлайн, ИИ хоста, модалки | Разнести по хукам/модулям при рефакторинге (см. фазу 4) |
| `src/App.tsx` | Маршрутизация экранов, сайд-эффекты входа/восстановления | Не смешивать с логикой стола |
| `src/game/rotateState.ts` | Поворот канонического стола «я снизу» | Стабильный контракт индексов слотов |
| `src/lib/supabase.ts` | Клиент, глобальные таймауты fetch | Влияет на «висящие» запросы |
| `src/game/ai.ts` + `personalAi.ts` + `aiLearning.ts` | ИИ и персонализация | Фичефлаги; не смешивать с сетевым слоем |
| `src/lib/featureFlags.ts` | Включение экспериментов | Явно документировать, что в проде вкл/выкл |

Документы `docs/TURN-TIMEOUT-*.md` и др. — **описание идей**, не обязаны совпадать с кодом; после чистки либо обновить, либо пометить «устарело».

## 4. План работ по фазам (планомерно)

### Фаза A — Зафиксировать эталон (0.5–1 день)

1. Записать **точный коммит/ветку прода** (или `main` на дату деплоя).
2. `git diff main...HEAD` по файлам из таблицы выше — список **намеренных** фич vs **случайных** правок.
3. Прогнать сценарий: создать комнату → 3–4 игрока → торги → 2–3 взятки → вкладка в фон → возврат (зафиксировать, что именно «зависает»).

### Фаза B — Упростить слой слияния (ядро стабильности)

1. В `OnlineGameContext`: одна страница комментария **дерева решений**: waiting → всегда полная строка; playing → когда доверять `updated_at`, когда `isServerStateNewerOrEqual`, когда не трогать `game_state`.
2. Убрать дубли: например, опрос и Realtime не должны по-разному обходить одни и те же проверки (сейчас частично так).
3. После успешного `updateRoomState` — **один** путь обновления `lastSeen` + state (уже завязан на `select` из БД).

### Фаза C — Сессия и гонки

1. `performSessionRestore` + `leaveRoom` + `joinRoom`: инвариант «не применять старый `saved.roomId`, если в storage уже другой» — оставить, покрыть комментарием.
2. Не плодить вторые источники `roomId` в ref без таблицы «кто пишет».

### Фаза D — UI и размер `GameTable`

1. Вынести онлайн-специфику в `hooks/useOnlineTable.ts` или `game/online/` по мере необходимости (не обязательно сразу).
2. Оставить в компоненте только рендер и короткие обработчики.

### Фаза E — Документация и мёртвый код

1. Удалить неиспользуемые экспорты, неиспользуемые константы из старых таймаутов.
2. Обновить `AGENTS.md` / этот файл ссылкой «карта онлайна».

## 5. Критерий «готово»

- Один понятный сценарий merge для `game_state` в игре.
- Нет модалок/таймеров, обещающих автозамену без явного действия пользователя (кроме ручной паузы).
- Сравнение с `main`: либо осознанный merge в PR с описанием, либо откат отдельных файлов к `main` с повторным внесением только нужных фиксов.

## 6. Ограничения проекта (не сломать при чистке)

- Профиль: `getPlayerProfile` / `savePlayerProfile` — единственный источник имени/аватара.
- Мобильная вёрстка: правки ПК только в `@media (min-width: 1025px)` / `.game-table-root:not(.viewport-mobile)` — см. `docs/PC-CARDS-PROTECT-MOBILE.md`.

После прохождения фаз A–B обычно исчезает ощущение «нагромождения»: остаётся меньше взаимоисключающих веток в одном файле.
